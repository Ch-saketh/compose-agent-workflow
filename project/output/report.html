🎯 Problem Statement

Evaluating third-party SaaS applications for integration potential is traditionally a highly manual and time-consuming process. Product Operations engineers and Developer Advocates often spend hours digging through disparate developer portals to uncover authentication methods, API limitations, and Multi-Cloud Platform (MCP) support.

This manual approach doesn't scale when a platform needs to rapidly assess hundreds of potential integration targets to build a comprehensive agentic ecosystem.

📋 Assignment Objective

The core objective of this assignment was to build an autonomous pipeline capable of researching 100 SaaS applications, analyzing their API documentation, and determining their "Agent Buildability."

To succeed, the system needed to act as a resilient data pipeline that not only queries large language models (LLMs) but also cross-verifies the extracted data for accuracy, ensuring that downstream engineering decisions are based on high-confidence insights.

💡 Solution Overview

I architected a modular, multi-agent AI pipeline driven by Google's Gemini Pro. Rather than attempting a massive, brittle zero-shot prompt, the system relies on specialized, decoupled agents.

  • CSV-Driven Data Layer: A simple apps.csv acts as the single source of truth, allowing non-technical stakeholders to queue new applications effortlessly without touching the codebase.
  • JSON Intermediate State: By enforcing strict JSON schema outputs from the LLM, the pipeline caches raw data locally. This prevents data loss on failures, avoids redundant API calls, and allows the final presentation layer to be rebuilt instantly.
  • Isolated Verification: Recognizing that LLMs occasionally hallucinate URLs or hallucinate API features, a dedicated Verification Agent audits the initial research.
  • Decoupled Analyzer & HTML Deliverable: Analytics are separated from data gathering. Generating a static HTML file ensures the final report is portable, zero-dependency, and instantly deployable to any static hosting provider.

🤖 Agent Responsibilities

1. Discovery Agent

Identifies the canonical developer documentation, API references, and pricing pages to anchor the research in factual sources.

2. Research Agent

Crawls the discovered URLs and extracts structured information strictly adhering to the expected JSON schema.

3. Verification Agent

Acts as an adversarial auditor. It scores the confidence of the research and applies automated corrections if assumptions were made.

4. Analyzer

Aggregates the individual verified JSON files to compute pipeline-wide metrics, distributions, and key insights.

🚧 Challenges Encountered

  • API Rate Limits: The free tier of the Gemini API imposes strict rate limits (20 requests per day). I mitigated this by implementing local JSON caching. If an API call fails with a 429 error, the pipeline gracefully skips the app without destroying previously gathered data.
  • Hallucinated URLs: Early iterations of the LLM occasionally hallucinated documentation links. This was resolved by creating the adversarial Verification Agent, which cross-checks the URLs and applies corrections before saving the final state.

🚀 Future Improvements

  • Parallel Processing: Implement asyncio to process multiple applications concurrently, drastically reducing pipeline execution time (assuming higher API quotas).
  • Headless Browsing: Integrate Playwright or Selenium to scrape developer portals that require JavaScript rendering before passing the raw HTML to the LLM.
  • Vector Database Integration: Embed the discovered API documentation into a vector database (e.g., Pinecone) to allow engineers to chat directly with the research dataset.
